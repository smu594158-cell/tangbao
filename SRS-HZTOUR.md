## 1. 业务目标与核心场景

### 1.1 编写目的
本文档旨在明确“AI 杭州旅游助手”系统的需求规格，为系统的设计、开发、测试和验收提供依据。本文档严格遵循 GB/T 8567-2020《计算机软件需求规格说明规范》。

### 1.2 业务目标
*   **C 端用户痛点解决：** 针对普通游客，提供一站式解决攻略繁琐问题的功能，减少用户在多个平台之间切换的时间和精力成本。
*   **毕业设计评审亮点：** 突出大模型本地部署（Ollama）、MCP（Model Context Protocol）的创新应用，以及支持高并发的微服务架构设计，展示扎实的技术功底与前沿技术视野。

### 1.3 核心场景
系统聚焦于“杭州区域”的旅游服务。
*   用户通过文本与“杭州导游”角色的 AI 进行多轮对话，获取个性化的旅游建议。
*   系统通过整合高德地图数据、小红书等平台内容，智能生成高质量的旅游推文和景点介绍。
*   用户可获取包括步行、公交、驾车、骑行在内的多种出行路径规划，并在地图上进行沉浸式交互预览。

## 2. 功能需求拆解

### 2.1 智能问答模块
*   **交互形式：** 仅支持纯文本交互，暂不考虑语音和图片输入。
*   **AI 角色设定：** “杭州导游”，系统需内置相应的 System Prompt 以约束其回答风格和领域边界。
*   **技术实现：** 后端（eino）调用本地部署的 Ollama 服务（模型：qwen:4b）。
*   **性能指标：** 支持多轮上下文，平均响应延迟 < 800 ms，回答准确率 ≥ 85%。

### 2.2 景点资讯与推文生成器
*   **数据来源：** 结合系统内部的景点数据库及模拟爬取的小红书等内容数据。
*   **功能描述：** 支持用户通过点击预设热门景点或自定义输入任意地点名称，一键生成包含【景点特色】、【最佳游览时间】、【交通提示】三大要素的高质量旅游推文。
*   **生成质量：** 文本风格要求生动、适合社交媒体（如小红书），原创度检测 ≥ 70%，生成字数支持在 200–400 字之间可调。

### 2.3 交互式地图与路线规划 (高德地图 MCP 集成)
*   **架构设计：** 后端部署本地 MCP Server 代理，封装高德地图 Web 服务 API 进行地理编码和路线计算；前端基于高德地图 JS API v2.0 实现沉浸式渲染。
*   **功能支持：** 提供 POI（兴趣点）搜索、点击地图自动逆地理编码，以及全面的路径规划（支持步行、公交、驾车、骑行四种方式），并支持一键唤起高德 App 或 H5 导航。
*   **性能指标：** 接口 QPS ≥ 50，99 分位（P99）延迟 < 200 ms。

## 3. 非功能需求

### 3.1 技术选型与系统架构模块划分
系统基于领域驱动设计 (DDD) 划分为以下 5 个核心模块（限界上下文）：
1. **IAM 模块 (身份与访问管理)**：负责用户注册、登录、密码加盐脱敏、JWT 鉴权及 RBAC 角色控制。
2. **Chat 模块 (AI 智能问答域)**：负责多轮对话上下文管理，注入“杭州导游”人设，调用大模型并异步落库。
3. **Tour & Content 模块 (旅游与内容域)**：负责景点基础信息管理，处理用户生成推文的请求，调度本地大模型进行内容生成。
4. **Geo 模块 (高德地图 MCP 集成域)**：代理封装高德地图 API，提供 POI 搜索与多种出行方式的路径规划。
5. **Frontend 模块 (前端展示域)**：纯 SPA 应用（React 18 + Vite），负责与 C 端用户的直接交互，包括聊天气泡界面、高德地图组件展示等。

*   **编程语言：** Go 1.25.0
*   **Web 框架：** Gin 框架（提供 RESTful API）。
*   **数据库：** MySQL 8.0
*   **AI 模型服务：** Ollama 本地部署，运行 qwen:4b 模型。
*   **第三方服务：** 高德地图 Web 服务 API (后端) & JS API (前端)。

### 3.2 性能需求
*   **并发能力：** 单实例需支持 500 并发连接。
*   **资源限制：** 仅针对 Go 后端服务本身：CPU 利用率 ≤ 70%，内存占用 ≤ 2 GB。

### 3.3 安全与隐私
*   **常规防护：** 实现防 SQL 注入、XSS、CSRF 等常见 Web 漏洞的安全策略。
*   **内容合规：** 内置敏感词过滤机制，防止 AI 输出或爬取违规内容。
*   **数据脱敏：** 用户的密码等敏感信息必须经过加盐哈希等脱敏/加密处理后方可存储入库。
*   **API 密钥隔离：** 前端 JSAPI 密钥与后端 Web 服务密钥必须严格分离，避免越权调用。

### 3.4 可维护性
*   **架构分层：** 采用标准的模块化分层架构（如 `api/http`, `usecase`, `repository` 层），降低代码耦合度。
*   **测试覆盖率：** 核心逻辑单元测试代码覆盖率 ≥ 80%。
*   **接口文档：** 提供 Swagger UI 或 Markdown 格式 API 文档，确保前后端联调顺畅。

## 4. 数据模型与接口契约

### 4.1 核心数据模型
系统主要包含以下数据实体表（基于 MySQL 8.0）：
1.  **用户表 (users)：** 存储用户基础信息，支持传统的“账号 + 密码”注册与登录机制（密码需哈希加密）。
2.  **景区表 (attractions)：** 存储杭州各景点的基础信息（名称、位置、简介等）。
3.  **聊天记录表 (chat_histories)：** 存储用户与 AI 的多轮对话历史。策略：仅保留最近 30 天的记录，系统需包含定时清理机制。
4.  **生成文本表 (generated_texts)：** 存储基于大模型生成的景点/人物介绍文本、推文及相关元数据。

### 4.2 接口契约规范
*   **通信协议：** 前后端调用统一通过 HTTP 访问 RESTful 接口。
*   **请求/响应规范：** RESTful 接口需提供明确的 JSON 请求和响应字段定义。
*   **错误码处理：** 制定全局统一的业务错误码字典，规范错误响应结构（如 `{"code": 40001, "message": "密码错误", "data": null}`）。
*   **分页策略：** 针对列表查询接口（如查询景点列表），采用基于 `page`（页码）和 `size`（每页条数）的传统 Offset 分页策略。

## 5. 部署与运维

### 5.1 容器化部署
*   提供完整的 `docker-compose.yml` 文件，支持一键拉起基础设施栈，包括：MySQL 数据库、Ollama 服务、以及日志监控组件。
*   Go 后端服务与 React 前端应用支持本地独立编译与运行。

### 5.2 日志与监控
*   集成 Loki + Promtail 方案进行分布式日志收集与聚合。
*   配置 Grafana 看板，实现日志与系统监控指标的可视化展示（提升毕设演示效果）。

### 5.3 CI/CD 与工程化
*   提供标准的 `Makefile` 或 npm scripts，封装常用构建、测试、运行命令。
*   **版本控制：** 仅需在本地进行代码版本管理即可，无需将项目上传至 GitHub 等远程代码托管平台，本地运行展示即满足毕设要求。

## 6. 验收标准

### 6.1 功能验收
*   **智能问答：** 使用预设的 20 组真实杭州旅游相关问题进行测试，AI 回答准确率需 ≥ 85%。
*   **文本生成：** 生成的旅游推文必须包含“景点特色”、“最佳游览时间”和“交通提示”三大要素，语言生动，通过主流 Plagiarism（查重）工具检测，原创度达标。
*   **地图与导航：** 前端能够基于高德 JSAPI 正常渲染地图、绘制路线（支持自适应视野缩放），且能处理跨城公交等复杂场景。

### 6.2 性能验收
*   *注：根据业务沟通，本项目暂不执行严格的性能压测。* 重点验收系统在功能层面的稳定性和本地大模型推理资源消耗的合理性（单次推文生成 < 2s）。
